We approached this challenge by trying two pipelines in parallel. One went heavy on feature engineering (170+ engineered features, multi-target encoding, KNN target features, multi-seed stacking, pseudo-labeling) and topped out at 0.8407. The other went the opposite direction: a small set of carefully selected features fed into stacked Optuna-tuned ensembles, which reached 0.8633. This report walks through the second pipeline, which became our submitted solution. The heavy-engineering branch is preserved in \texttt{code/experiments/our\_pipeline/} for traceability and is referenced in the Section 3 comparison.

\section{Data exploration and preprocessing}
\label{sec:eda}

\subsection*{What the data looks like}

35{,}000 training rows, 15{,}000 test rows, 55 features. Two targets: \textbf{RiskTier} (categorical, 0 through 4) and \textbf{InterestRate} (continuous, 4.99\% to 35.99\%). The Kaggle score weighs them equally.

A few things jumped out as soon as we started exploring (Figure~\ref{fig:targets}).

RiskTier is balanced. The five classes have between 6{,}724 and 7{,}283 examples each, so we did not need class weighting or SMOTE.

InterestRate is harder than the headline statistics suggest. The mean is 7.31\%, the median is 6.08\%, but a third of all rows sit exactly at 4.99\%, the legal floor. Tiers 0 through 3 all live in the same narrow band: mean rates between 6.0\% and 6.6\%, standard deviations around 1.5. Tier 4 is a different population. Mean 11.55\%, standard deviation 7.36, and the right tail goes all the way to 35.99\%. The two-regime structure (prime versus subprime) shaped a lot of what came later.

We did the variance decomposition early because we wanted to know how much $R^2$ we could buy from getting the tier right. Within-tier variance accounts for 72\% of total rate variance; perfect tier prediction caps $R^2$ at about 0.28. Most of the rate signal comes from features other than tier, which is why we eventually fed back the OOF tier probabilities into the regressor (Section 2).

\begin{figure}[h]
    \centering
    \includegraphics[width=\linewidth]{fig_targets.pdf}
    \caption{Left: RiskTier is roughly balanced across the five classes. Middle: InterestRate has a heavy spike at the 4.99\% floor and a long right tail. Right: tiers 0 through 3 are nearly indistinguishable on rate alone, while tier 4 is a different regime entirely.}

\end{figure}

\subsection*{Smart preprocessing — driven by EDA, not by reflex}

The big lesson on this dataset was that overzealous handling of missing values and categoricals adds more noise than signal. We let the EDA tell us what to do, column by column.

\textbf{Structural-zero columns.} Eight columns are missing because the borrower genuinely has no such asset or liability: \texttt{PropertyValue} and \texttt{MortgageOutstandingBalance} for renters, \texttt{StudentLoanOutstandingBalance} for older applicants, \texttt{CollateralValue} for unsecured loans, \texttt{VehicleValue}, \texttt{InvestmentPortfolioValue}, \texttt{AutoLoanOutstandingBalance}, \texttt{SecondaryMonthlyIncome}. We fill these with zero. Median imputation here would invent a typical-borrower mortgage for a renter, which is wrong.

\textbf{Selective missingness flags.} Only two columns showed a meaningful target shift between ``missing'' and ``present'': \texttt{InvestmentPortfolioValue} and \texttt{RevolvingUtilizationRate}. We added \texttt{\_isMissing} flags for those two, and for nothing else. The heavy-engineering branch added a flag for every column with NaN; that turned out to add 12 noisy binary columns the model had to learn to ignore.

\textbf{Ordinal encoding for EducationLevel.} EDA showed the mean RiskTier walks monotonically from 2.17 (No Diploma) to 1.82 (PhD). We mapped to integers 0 through 5 to preserve that ordering, instead of one-hot which would erase it.

\textbf{Drop low-signal categoricals.} \texttt{State}, \texttt{JobCategory}, and \texttt{MaritalStatus} all had nearly flat target means across their levels. Including them just gave the trees more random splits to chase. We dropped them entirely. \texttt{CollateralType} stayed (signal was real, NaN replaced with the explicit token \texttt{None}).

\begin{figure}[h]

    \caption{Columns with more than 1\% missingness. Each block lines up with a structural sub-population. Most missingness flags add noise (the model can already see the source column); we kept flags only where EDA showed a target shift.}

\end{figure}

\section{Feature engineering}
\label{sec:feateng}

\subsection*{Five engineered features, not fifty}

We deliberately stayed restrained on engineered features. The five we kept all encode something a real underwriter would compute:

\textbf{TotalLatePayments} = Late30 + Late60 + Late90. A simple sum of all late-payment events, regardless of severity.

\textbf{DerogMarks} = ChargeOffs + Collections + Bankruptcies + PublicRecords. The count of derogatory marks on file.

\textbf{SatisfactoryAccountRatio} = NumberOfSatisfactoryAccounts / (NumberOfOpenAccounts + 1). The fraction of accounts in good standing — a true non-linear feature (division) the model would otherwise have to learn.

\textbf{RiskSeverityScore} = $1\!\cdot\!$Late30 $+ 3\!\cdot\!$Late60 $+ 9\!\cdot\!$Late90 $+ 15\!\cdot\!$ChargeOffs $+ 10\!\cdot\!$Collections $+ 30\!\cdot\!$Bankruptcies $+ 8\!\cdot\!$PublicRecords. A domain-weighted severity composite that penalises worse marks more heavily, in the spirit of FICO.

These five together with the raw 55 features and the 2 missingness flags give 62 model inputs going into selection.

\subsection*{Four-signal feature selection}

This is where the design diverges from the heavy-engineering branch and where most of the score came from. We score every feature on four orthogonal signals and drop a feature only if it scores low on all four:

\begin{enumerate}
\item XGBoost gain importance for RiskTier (300-tree XGB, default reg).
\item XGBoost gain importance for InterestRate.
\item Mutual information vs RiskTier (\texttt{sklearn.mutual\_info\_classif}).
\item Mutual information vs InterestRate (\texttt{mutual\_info\_regression}).
\end{enumerate}

The four-signal consensus matters because single-signal selection kept killing borderline-useful features. A feature that XGBoost ignores but mutual information picks up is often something the linear meta-learner uses; a feature both methods rank low is genuinely noise. After this step, 38 features survive.

We then prune redundancy: for every pair of features with $|\rho| > 0.95$, drop whichever has the lower max-signal across the four scorers. That removes another 20, leaving the final \textbf{18 features}. Figure~\ref{fig:reduction} shows the funnel.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.75\linewidth]{fig_feature_reduction.pdf}
    \caption{The selection funnel. We start at 55 raw columns, add 5 domain aggregates and 2 missingness flags (62 total), drop 24 with the four-signal consensus, then 20 redundant by correlation pruning. Final input to the modelling stage: 18 features.}
    \label{fig:reduction}
\end{figure}

\subsection*{Cross-task signal: tier probabilities into the regressor}

Since perfect tier prediction caps $R^2$ at only 0.28, the regressor needs more than just the tier label. We use the classifier's full distribution. Five OOF probabilities (one per tier, from a 5-fold \texttt{cross\_val\_predict}) plus a soft \texttt{expected\_tier} = $\sum_i p_i \cdot i$ are added to the regressor's feature set. Because they are out-of-fold, no row sees a model that was trained on it.

This single change buys about $+0.04$ on the regressor's $R^2$ in 5-fold CV.

\section{Model selection and tuning}
\label{sec:models}

\subsection*{Six tuned base learners per task}

The base set is six models per task, each tuned with Optuna's TPE sampler on 3-fold CV: \texttt{xgb1} (25 trials), \texttt{xgb2} (25), Random Forest (15), Extra Trees (15), HistGradientBoosting (20), MLPClassifier / MLPRegressor (20). The two XGB slots use deliberately different search spaces (one shallow-and-many, one deep-and-fewer) to reduce error correlation in the stack. The MLP feeds standardised inputs with one of three scalers (\texttt{StandardScaler}, \texttt{MinMaxScaler}, \texttt{RobustScaler}) selected as a hyperparameter.

\subsection*{Stacked ensembles}

\textbf{Task A.} \texttt{StackingClassifier} from scikit-learn: six tuned base classifiers feed predicted probabilities to a \texttt{LogisticRegression} (multinomial, $C=1$) meta-learner, with internal 3-fold CV to generate the meta features. Outer 5-fold StratifiedKFold for the reported score: $88.12\% \pm 0.87\%$ accuracy.

\textbf{Task B.} \texttt{StackingRegressor}: same six base regressors trained on the augmented feature set (18 selected + 5 OOF tier probabilities + 1 expected\_tier = 24 features), with \texttt{Ridge}($\alpha=1$) as the meta-learner. Outer 5-fold KFold: $R^2 = 0.8453 \pm 0.0140$, RMSE = $1.6417\% \pm 0.013\%$.

Combined: $0.5 \cdot 0.8812 + 0.5 \cdot 0.8453 = \mathbf{0.8633}$.

\subsection*{Overfitting controls}

\textbf{Per-fold CV throughout.} Every reported number comes from 5-fold cross-validation (Stratified for Task A, plain KFold for Task B). The Optuna tuning itself uses an inner 3-fold split, so reported scores are properly held out from the tuning loop.

\textbf{Cross-validated tier features.} The OOF tier probabilities fed into Task B come from \texttt{cross\_val\_predict} with the same outer fold structure, so each row's tier features come from a model that did not see that row. Without this, $R^2$ inflates by about 0.02 in CV and collapses on the held-out test.

\textbf{Trial budgets capped.} Per-model trial counts are 15--25, deliberately small. With six models the total search is 120 trials, enough to find a good neighbourhood but not so many that we overfit the inner 3-fold CV.

\textbf{Conservative meta-learners.} Logistic Regression with $C=1$ for Task A and Ridge with $\alpha=1$ for Task B. Both are linear and heavily regularised; the meta-learner's job is to weight the base learners, not to add capacity.

\subsection*{Did the same features matter for both tasks?}

Mostly yes. Both top-10 importance lists are dominated by the same five names: \texttt{RiskSeverityScore}, \texttt{DebtToIncomeRatio}, \texttt{RevolvingUtilizationRate}, \texttt{TotalLatePayments}, \texttt{NumberOfSatisfactoryAccounts}. After that the lists diverge in the way you would expect from the structure of the two problems. Task A weights structural binary signals more highly: \texttt{HomeOwnership=Mortgage}, \texttt{IncomeVerified}, \texttt{HasCoApplicant}. These cleanly separate tier boundaries. Task B leans on continuous leverage ratios: \texttt{LoanToIncomeRatio}, \texttt{PaymentToIncomeRatio}, \texttt{SatisfactoryAccountRatio}. That makes sense — Task B is pricing a rate within a tier, not picking the tier itself, so it cares about graded financial stress rather than categorical risk class.

The fact that the two tasks share so many top features is also why cross-task stacking works. The OOF tier probabilities collapse the joint information into six numbers the regressor can use directly.

\subsection*{The score progression}

Figure~\ref{fig:progression} shows how the combined CV score climbed step by step. Two interventions did most of the work: aggressive feature selection ($+0.07$) and stacking the tuned base learners ($+0.03$). Smart preprocessing was the hidden enabler — without it the rest of the pipeline runs on noisier inputs and the stacking gain shrinks.

\subsection*{Why this beat the heavy-engineering branch}

We ran a parallel pipeline (in \texttt{code/experiments/our\_pipeline/}) that went the opposite direction: 50 financially motivated engineered features, K-fold target encoding for high-cardinality categoricals, K-NN target features, multi-seed stage-2 LightGBM stacker, pseudo-labeling on high-confidence test rows. That pipeline took 14 iterations and topped out at \textbf{0.8407}. The canonical pipeline reached 0.8633 with about a third of the code and no pseudo-labeling.

Looking back, we think the heavy branch was drowning the signal. With 170 inputs the meta-learner has too many correlated features to weight properly and the base learners spend capacity learning to ignore noise. The 18-feature stack is leaner, faster (50 minutes versus 100), and leaves the boosting models room to actually fit the high-signal structure. The bitter lesson on this dataset: feature selection mattered more than feature engineering.

\subsection*{What we learned}

Three things, ranked by impact:
\begin{enumerate}
\item Aggressive feature selection beats aggressive feature engineering when the dataset is small (35k rows) and the natural feature count is modest (55 columns). Adding more features dilutes the signal that already exists.
\item Per-model Optuna tuning on a small budget (15--25 trials) is enough to make stacking useful. Stacking under-tuned base learners gains very little.
\item OOF cross-task features have to be computed with the same outer fold structure as the eval, otherwise CV optimism inflates by 0.02--0.03. We caught this early because both pipelines used 5-fold CV with the same seed.
\end{enumerate}
